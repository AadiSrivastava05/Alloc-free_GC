\title{\textbf{Alloc-free-gc: Design and Performance Evaluation of Allocation-conscious Semi-space Garbage Collectors}}
\author{Independent Project Report}
\date{\today}

\begin{document}
\maketitle

\begin{abstract}
\texttt{Alloc-free-gc} is an independent experimental garbage-collection runtime
for a small OCaml-style language implementation. The project evaluates whether
semi-space copying collectors written in OCaml/OxCaml can remain competitive
when the allocation fast path and heap ownership are handled by a compact C
runtime layer. The implementation compares five collector variants: a normal
OCaml collector, a zero-allocation OxCaml collector, an off-heap metadata
zero-allocation collector, a threaded off-heap collector, and a stack-threaded
collector service. The benchmark data show that the variants are close in
absolute runtime, with \texttt{zero\_offheap} giving the best aggregate mean in
the current benchmark matrix. However, the checked-in data contain one run per
configuration, so the results should be interpreted as directional evidence
rather than statistically conclusive proof. The main finding is that
zero-allocation and off-heap metadata techniques can make an OxCaml collector
competitive with a normal OCaml collector, but the benefit is workload-sensitive
and modest in the present implementation.
\end{abstract}

\section{Introduction}

Garbage collectors are normally implemented in low-level runtime code, because
they must manipulate raw heap words, update roots, copy objects, and avoid
accidentally allocating while memory is already under pressure. The goal of
\texttt{Alloc-free-gc} is to test a different engineering approach: keep the
allocation fast path and heap memory management in C, but implement the
collection logic in OCaml/OxCaml with careful attention to allocation behavior.

The project is self-contained. It defines its own runtime interface, benchmark
programs, build rules, collector implementations, result CSV files, and plotting
scripts. The runtime uses an OCaml-style value representation, but the report
treats \texttt{Alloc-free-gc} as an independent system rather than as an
extension of another repository.

The work is organized around three questions:

\begin{enumerate}
  \item Can a normal OCaml implementation of a copying collector serve as a
  reasonable baseline behind a C runtime ABI?
  \item Do OxCaml zero-allocation annotations improve the collection hot path in
  measurable end-to-end benchmarks?
  \item Does moving collector metadata off the OCaml heap, or running
  collection through a persistent worker thread, improve performance or
  stability?
\end{enumerate}

\section{System Overview}

\texttt{Alloc-free-gc} is split into three layers:

\begin{itemize}
  \item \textbf{Runtime layer}: \texttt{runtime.c} and \texttt{runtime.h}
  provide allocation entry points, tagged-value helpers, explicit root
  registration, mutator thread startup, and stop-the-world coordination.
  \item \textbf{FFI and heap layer}: \texttt{gc\_ffi.c} embeds the OCaml/OxCaml
  runtime, owns the semi-space heap, exposes the \texttt{mmtk\_*} ABI used by
  the runtime layer, and provides no-allocation C stubs for raw word access.
  \item \textbf{Collector layer}: one OCaml/OxCaml collector module is selected
  at build time using the \texttt{GC=} Makefile variable.
\end{itemize}

This separation is important for the benchmark interpretation. All variants use
the same C allocation fast path. Therefore, the benchmark primarily compares
collection and root-scanning implementation style, not allocation ABI overhead.

\section{Runtime Model}

\subsection{Value Representation}

The runtime uses a tagged machine-word representation. Integers have low bit
\(1\), while heap pointers have low bit \(0\). The helper macros in
\texttt{runtime.h} encode and decode integers:

\[
  \texttt{long2val}(x) = (x \ll 1) + 1, \qquad
  \texttt{val2long}(x) = x \gg 1.
\]

Heap object pointers exposed to mutator code point to the object body. The
object header is stored one word before the body pointer. The header stores the
object size and tag. During copying, the collector overwrites the old header
with a forwarding marker and stores the forwarded body pointer in the first
payload word.

\subsection{Heap Layout}

The heap is a two-space copying heap with a from-space and a to-space. Allocation
uses a bump pointer in from-space. When allocation fails, the runtime stops
mutators, invokes the selected collector, copies reachable objects to to-space,
swaps the spaces, and resets the bump pointer to the number of live bytes.

The configured heap size is defined in \texttt{runtime.h}. On a 64-bit target,
the current configuration gives approximately one GiB per semi-space:

\[
  \texttt{HEAP\_SIZE} = 4096 \times 1024 \times 32 \quad \text{words}.
\]

\subsection{Root Tracking}

Root tracking is explicit. Each mutator has:

\begin{itemize}
  \item a thread-local root stack,
  \item a stack-index slot,
  \item a return-value root slot, and
  \item an active/inactive state.
\end{itemize}

Local variables that may contain heap pointers are registered by address. During
collection, the collector reads each root value, copies the referenced object if
needed, and writes the updated pointer back through the registered address. This
keeps root discovery precise and avoids conservative stack scanning.

\subsection{Stop-the-world Coordination}

Collection is stop-the-world. When allocation cannot be satisfied, the
requesting mutator marks GC as requested and waits until active mutators reach a
polling or allocation point. After all mutators are stopped, collection runs and
the runtime broadcasts completion. This design avoids concurrent mutation during
copying, which keeps the collector implementation small enough to compare
implementation strategies directly.

\section{Collector Variants}

\subsection{Normal OCaml Collector}

\texttt{gc\_normal.ml} is the baseline implementation. It implements the
semi-space collector in ordinary OCaml, using OCaml arrays and references for
metadata and mutable state. It is intentionally straightforward:

\begin{enumerate}
  \item initialize \texttt{free\_ptr} and \texttt{scan\_ptr} to the beginning of
  to-space;
  \item copy static roots, mutator return roots, and local roots;
  \item perform Cheney-style breadth-first traversal over copied objects;
  \item copy pointer fields encountered during traversal; and
  \item finish collection by swapping spaces in the C heap layer.
\end{enumerate}

This variant establishes the cost of writing the collector in ordinary OCaml
without special zero-allocation restrictions.

\subsection{Zero-allocation OxCaml Collector}

\texttt{gc.ml}, built as \texttt{GC=zero}, rewrites the hot path with OxCaml
zero-allocation annotations. The collector avoids ordinary heap allocation in
bounded collection logic. For example, the copy function returns both the
possibly updated pointer and the next free pointer using an unboxed multi-value
style rather than allocating a pair or mutating an OCaml reference.

This design tests whether local hot-path allocation control is enough to improve
runtime while leaving persistent metadata in OCaml-managed arrays.

\subsection{Zero-allocation Off-heap Metadata Collector}

\texttt{gc\_zero\_offheap.ml}, built as \texttt{GC=zero\_offheap}, moves
persistent metadata out of the OCaml heap. Static root slots, root-stack
pointers, stack-index pointers, return-root pointers, active-thread flags, and
counters are stored in malloc-backed memory. The collector accesses this region
through no-allocation C stubs.

The motivation is to reduce interference between the collector implementation
and the embedded OCaml runtime. If collector metadata lives outside the OCaml
heap, then collection bookkeeping is less likely to create pressure on the host
runtime's own memory manager.

\subsection{Threaded Off-heap Collector}

\texttt{GC=zero\_offheap\_threaded} uses the same off-heap metadata collector
but compiles \texttt{gc\_ffi.c} with a persistent sleeping GC worker thread.
When allocation fails and mutators have stopped, the C runtime wakes the worker
to perform collection.

This variant tests whether keeping a long-lived OCaml-capable worker thread
reduces per-collection runtime coordination overhead. The tradeoff is an extra
synchronized command path between the mutator runtime and the GC worker.

\subsection{Stack-threaded Collector Service}

\texttt{GC=zero\_stack\_threaded} routes collection through a long-lived OxCaml
service loop implemented in \texttt{gc\_stack\_threaded.ml}. In this design, C
owns the mutator metadata, and the OxCaml service thread performs only the
collection logic when it receives a command.

This design is motivated by stack allocation: values allocated on a stack cannot
escape the region that owns them. A persistent service loop gives the collector
a long-lived execution region for repeated collection work.

\section{Benchmark Methodology}

\subsection{Workload}

The benchmark is an allocation-heavy binary-tree workload included in
\texttt{Alloc-free-gc/benchmarks}. Each worker builds:

\begin{itemize}
  \item one stretch tree of depth \(\texttt{max\_depth}+1\),
  \item one long-lived tree of depth \(\texttt{max\_depth}\), and
  \item many temporary trees at even depths from 4 to \(\texttt{max\_depth}\).
\end{itemize}

Each tree node has left and right child fields plus a pointer to a 128-word data
array. This creates a mixture of short-lived temporary objects and long-lived
reachable objects. Correctness is checked by traversing trees and computing
node-count checksums.

\subsection{Benchmark Matrix}

The main benchmark driver is
\texttt{Alloc-free-gc/benchmarking/run\_gc\_benchmark.sh}. The checked-in
result matrix uses:

\begin{itemize}
  \item depths: 11, 12, 13, and 14;
  \item mutator thread counts: 1 through 8;
  \item variants: \texttt{normal}, \texttt{zero}, \texttt{zero\_offheap},
  \texttt{zero\_offheap\_threaded}, and \texttt{zero\_stack\_threaded};
  \item repetitions: 1 per configuration.
\end{itemize}

The plotting script writes summary statistics and speedup ratios to
\texttt{Alloc-free-gc/benchmarking/benchmark\_plots}. Because the checked-in
data have \(n=1\) for each configuration, standard deviations are not meaningful.
The results are still useful for directional analysis, but strong claims require
additional repeated runs.

\subsection{Per-depth Observations}

At depth 11, runtimes are small and differences are noisy. The stack-threaded
and off-heap variants perform well in several configurations, but the baseline
normal collector remains competitive.

At depth 12, the best variant changes across thread counts. The threaded
off-heap collector is best at 2 and 7 threads, off-heap is best at 1 and 4
threads, and the plain zero-allocation collector is best at 6 threads. This
suggests that the overheads being measured are not dominated by one simple
factor.

At depth 13, the normal collector wins at 1, 3, and 5 threads, while the
threaded off-heap variant wins at 4, 6, and 8 threads. This is the clearest
evidence that no single implementation strategy dominates the benchmark.

At depth 14, the zero-allocation variants become more favorable at larger thread
counts. The best results at 5, 6, 7, and 8 threads are from
\texttt{zero\_stack\_threaded} or \texttt{zero\_offheap\_threaded}. At 8
threads, however, \texttt{zero\_stack\_threaded} is worse than normal, showing
that service-thread coordination can also regress.

\section{Analysis}

\subsection{Zero-allocation Alone Is Not a Universal Win}

The plain \texttt{zero} variant has a slightly better aggregate mean than
\texttt{normal}, but it is faster in exactly half of the measured configurations.
This means that zero-allocation annotations on the collection hot path are not
by themselves enough to dominate ordinary OCaml. The likely reason is that total
elapsed time includes more than OCaml allocation in the collector: root scanning,
raw memory loads and stores through C stubs, object copying bandwidth, runtime
coordination, and benchmark scheduling all contribute.

\subsection{Off-heap Metadata Is the Most Promising Change}

\texttt{zero\_offheap} is the most consistent improvement in the checked-in
matrix. It is faster than \texttt{normal} in 21 of 32 configurations and has the
lowest mean runtime. This supports the design hypothesis that persistent
collector metadata should not necessarily live in the embedded OCaml heap.

The improvement is still modest. Off-heap metadata reduces one source of
overhead, but the collector still performs the same fundamental work: scan
roots, copy objects, traverse reachable fields, and update pointers.

\subsection{Threaded Collection Has Mixed Results}

\texttt{zero\_offheap\_threaded} is close to \texttt{zero\_offheap} in aggregate
mean and is best in several larger configurations. This suggests that a
persistent worker can help when per-collection runtime coordination would
otherwise be costly.

\texttt{zero\_stack\_threaded} is also often competitive, but it has the worst
single result at depth 14 and 8 threads. That outlier matters: the service-thread
design adds synchronization, and that synchronization can outweigh the benefit
of stable stack-local collector state.

\subsection{Meaningful Result}

The meaningful result is not that one collector variant wins everywhere. The
stronger conclusion is that an allocation-conscious OxCaml collector can be made
competitive with an ordinary OCaml collector behind a C runtime ABI. Off-heap
metadata appears to be the most useful optimization among the tested variants,
while persistent worker-thread designs need more measurement before their value
is clear.

\section{Threats to Validity}

\begin{itemize}
  \item Each checked-in configuration has one run. This prevents confidence
  intervals and makes outliers difficult to classify.
  \item The benchmark is one allocation-heavy binary-tree workload. Other
  allocation patterns may produce different results.
  \item The results measure total elapsed time, not isolated GC pause time.
  Mutator work, scheduler behavior, and synchronization overhead are included.
  \item Some variants differ in runtime coordination strategy as well as
  collector metadata placement, so not every timing difference can be attributed
  to one isolated mechanism.
  \item The benchmark should be repeated with at least 5--10 runs per
  configuration before making strong quantitative claims.
\end{itemize}

\section{Conclusion}

\texttt{Alloc-free-gc} demonstrates that semi-space collection logic can be
implemented in OCaml/OxCaml while remaining competitive in an allocation-heavy
runtime benchmark. The normal OCaml baseline is already strong, and
zero-allocation annotations alone do not guarantee a clear speedup. The most
promising design is \texttt{zero\_offheap}, which stores persistent collector
metadata outside the OCaml heap and achieves the best aggregate mean in the
current result set.

The current evidence supports continued development, but not a final
performance claim. The next evaluation step should be repeated benchmarking with
variance reporting and, ideally, separate instrumentation for total elapsed
time, collection count, copied bytes, and GC pause time.
